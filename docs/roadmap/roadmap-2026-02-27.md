# Roadmap 2026 统一版（基于 README 目标 + 现有 roadmap 综合）

更新时间：2026-02-27

> 本文为“新的统一路线图”，用于替代多处状态分散。目标：把 README 的 Core Objective、当前 `docs/roadmap/roadmap.md` 的执行状态与优先级做一次收敛，并给出 3 套可选路线 + 推荐方案。

---

## 1. 核心目标（来自 README 2026 refresh）

打造 **personal local agent**，最大化单人判断力杠杆、最小化注意力成本：

- **Attention-saving first**：高信号、低噪音的状态/结论输出
- **Judgment leverage**：agent 负责探索/执行，人类专注目标与决策
- **Proactive + overrideable**：主动行动但保留明确用户权威
- **Subagent leverage**：并行探索 → 收敛决策输出
- **Context compression**：长跑场景不降质

**北极星场景：Calendar + Tasks 闭环（M0）**

## 4. 三套路线方案（含影响 / 成本 / 风险 / 回滚）

### 方案 A（推荐）：**可靠性闭环优先**
**顺序：Coding Gateway → Steward Reliability → Planning/Memory → Eval**

- **影响**：最高（解决能力入口 + 长链路稳定性）
- **成本**：中（多模块联动）
- **风险**：低-中（连续可验收）
- **回滚**：低（分阶段可停）

**适用**：追求稳态能力、避免“能力点多但不可靠”的系统崩溃。

---

### 方案 B：**产品闭环优先（Calendar/Tasks 再扩展）**
**顺序：Calendar/Tasks UX/Approval/Scheduler → Eval → Coding Gateway → Planning/Memory**

- **影响**：中（强化北极星体验，但长期扩展性不足）
- **成本**：中
- **风险**：中（长期可能形成局部优化）
- **回滚**：低

**适用**：要快速拉齐外部体验，争取用户/展示价值。

---

### 方案 C：**平台/生态优先（外部 agent + 多平台扩展）**
**顺序：External orchestration → Web/CLI 统一事件 → 平台化扩展**

- **影响**：低-中（非当前 NSM 核心）
- **成本**：高
- **风险**：高（依赖链重、回报慢）
- **回滚**：中

**适用**：如果当前阶段不是“单人判断力杠杆”，而是“生态扩张”。

---

## 5. 推荐方案（A）+ 执行切分

### P0（立即）— Coding Gateway Foundation
**DoD**：
- Gateway 合约稳定（Submit/Stream/Cancel/Status）
- 至少 1 个 adapter 注册成功
- CLI auto-detect 正常

### P1（本周）— Steward Reliability Closure
**DoD**：
- activation enforcement 有效
- evidence ref enforcement 形成闭环
- overflow 压缩保护关键 state
- L3/L4 approval UX 有 rollback + alternatives

### P1（本周-下周）— Planning + Memory Core
**DoD**：
- replan 最小 DAG 可上线
- memory D5 迁移稳定、无数据丢失

### P1（下周）— Evaluation Automation
**DoD**：
- 自动评测 pipeline + judge 接入 + report artifact

## 6. 交付节奏（建议）

- **Week 1**：Coding Gateway + Steward Reliability
- **Week 2**：Planning/Memory + Eval automation
- **Week 3**：验证 + 回归 + 产出报告

---

## 7. 推荐结论（简明版）

**走方案 A。**先把入口能力与跨轮可靠性闭环做扎实，再补规划/记忆与评测自动化。

这是最能保障“单人判断力杠杆”长期稳定的路径，风险可控，验收清晰。
