# Roadmap 2026 统一版（基于 README 目标 + 现有 roadmap 综合）

更新时间：2026-03-09

> 本文为“新的统一路线图”，用于替代多处状态分散。目标：把 README 的 Core Objective、当前 `docs/roadmap/roadmap.md` 的执行状态与优先级做一次收敛，并给出 3 套可选路线 + 推荐方案。

---

## 1. 核心目标（来自 README 2026 refresh）

打造 **personal local agent**，最大化单人判断力杠杆、最小化注意力成本：

- **Attention-saving first**：高信号、低噪音的状态/结论输出
- **Judgment leverage**：agent 负责探索/执行，人类专注目标与决策
- **Proactive + overrideable**：主动行动但保留明确用户权威
- **Subagent leverage**：并行探索 → 收敛决策输出
- **Context compression**：长跑场景不降质

**北极星场景：Calendar + Tasks 闭环（M0）**

## 4. 三套路线方案（含影响 / 成本 / 风险 / 回滚）

### 方案 A（推荐）：**CLI Runtime 收敛优先**
**顺序：CLI Runtime + Kaku → Session/Memory/Eval → Hooks/Scheduler → Panel/Team Recipe**

- **影响**：最高（先统一执行底座，再补自动化与产品层）
- **成本**：中（多模块联动）
- **风险**：低-中（连续可验收）
- **回滚**：低（分阶段可停）

**适用**：先把 runtime/session 能力做实，再把 hooks、调度、面板叠上去。

---

### 方案 B：**产品闭环优先（提醒/调度先开）**
**顺序：Scheduler/Reminder/Outreach → Eval → Runtime → Panel**

- **影响**：中（强化北极星体验，但长期扩展性不足）
- **成本**：中
- **风险**：中（长期可能形成局部优化）
- **回滚**：低

**适用**：如果目标是尽快把主动提醒与调度体验跑起来。

---

### 方案 C：**平台/生态优先（成员扩展先行）**
**顺序：AnyGen/Colab/member 扩展 → Panel → 更大生态**

- **影响**：低-中（非当前 NSM 核心）
- **成本**：高
- **风险**：高（依赖链重、回报慢）
- **回滚**：中

**适用**：如果当前阶段目标从 runtime 收敛转成生态扩张。

---

## 5. 推荐方案（A）+ 执行切分

### P0（立即）— CLI Runtime + Kaku Foundation
**DoD**：

- `MemberAdapter` 合约稳定
- 至少 1 个 member 能通过统一 runtime 跑通

### P1（本周）— Runtime Operability + Memory/Eval Closure
**DoD**：
- snapshot/context introspection 不再只暴露 TODO gap

### P2（下周）— Hooks + Scheduler Closure
**DoD**：
- hooks 事件可驱动 session continuation

### P3（后续）— Panel / Team Recipe
**DoD**：
- 一个界面看清多 session 状态
- team 退化为 recipe，而不是 runtime 本体

**走方案 A。**先把 CLI runtime 统一起来，再补 memory/eval 与 hooks/scheduler，最后再做 panel/team recipe。

这样更贴近当前代码与 `kaku` 方案，且比继续沿用旧的 Coding Gateway / Steward / Replan 队列更一致。
